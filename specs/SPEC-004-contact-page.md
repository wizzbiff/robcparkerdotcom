## Summary

Build the Contact page (`contact.html`) replacing the current placeholder. The page delivers a working contact form backed by Formspree, protected by a Cloudflare Turnstile challenge and a honeypot, plus two alternate contact channels (LinkedIn, direct email). It is the primary "reply path" for recruiters, hiring managers, and advisory inquiries generated by the home/About/Resume pages.

## Context

SPEC-001 established the site foundation and scaffolded a client-side validation pattern in `js/main.js` (`initContactForm`) whose submit handler currently stubs out backend wiring with a console message and an explicit "SPEC-004 will wire this up" comment. SPEC-002 (About) and SPEC-003 (Resume) both terminate in CTAs that point here, so this page is the conversion point for the whole site's hiring-funnel strategy.

The placeholder page carries `noindex`, ships only a heading + "coming soon" paragraph, and links back to home. All shared chrome (nav, footer, skip-link, styles) is already wired.

This is the **first SPEC that introduces external service dependencies** for this project. Two third-party services are bound in: Formspree (form backend + delivery) and Cloudflare Turnstile (bot defense). Formspree announced a native Turnstile integration in November 2025; the backend verifies the `cf-turnstile-response` token automatically once the Secret Key is pasted into the Formspree dashboard.

- Formspree × Turnstile announcement: https://formspree.io/blog/announcing-turnstile-integration/
- Formspree × Turnstile setup guide: https://help.formspree.io/hc/en-us/articles/46614263949331-Protecting-your-Forms-with-Cloudflare-Turnstile
- Cloudflare Turnstile docs: https://developers.cloudflare.com/turnstile/get-started/

## Requirements

### Page Structure

The page flows in this order:

1. **Hero / Page Intro** — Heading + short framing paragraph setting audience expectations and response-time norms
2. **Contact Form** — Primary CTA; name, email, message + Turnstile widget + honeypot
3. **Alternate Channels** — LinkedIn link + direct email address, for users who prefer not to use the form
4. **Response Expectations** — Single-line note on what happens next (merged into hero intro or rendered as a small aside; design call at implementation)

No headshot or new images. No CTA section at the bottom — this IS the conversion page; there is nowhere to send the user next. On success, the user stays on the page and sees an inline confirmation.

### Section 1: Hero / Page Intro

- **Heading:** H1 (e.g., "Let's talk" or "Get in touch" — marketing-copywriter to refine)
- **Framing paragraph:** 2-3 sentences. Explicitly welcomes Director/VP conversations AND advisory inquiries, without enumerating every possible reason. Short.
- **Response expectation sentence:** Sets expectation ("I reply within a few business days") — keeps expectations realistic rather than promising 24h turnaround
- **Layout:** Reuse the existing hero pattern variants established in SPEC-001/002. Text-only hero — no photo, no stats

### Section 2: Contact Form

**Fields (all required unless noted):**

| Field | Type | Validation | Notes |
|-------|------|-----------|-------|
| Name | `text` | Non-empty, trimmed | `autocomplete="name"` |
| Email | `email` | Valid email per existing regex | `autocomplete="email"` |
| Message | `textarea` | Non-empty, trimmed, min 10 chars | 6 rows, vertical resize only |
| `_gotcha` | hidden text | Must be empty on submit | Formspree honeypot convention |
| `cf-turnstile-response` | hidden (auto) | Populated by Turnstile widget | Verified by Formspree |

**Submission flow:**

- Form uses `method="POST"` to `https://formspree.io/f/{FORMSPREE_FORM_ID}`
- Submission is **AJAX** via `fetch()` with `Accept: application/json` — user stays on page
- Existing `initContactForm` in `js/main.js` is the integration point; the stub submit handler is replaced with the real fetch call
- Submit button is **disabled until the Turnstile widget has produced a token** (or disabled at render and re-enabled via the widget's `callback`). This prevents empty-token submissions reaching Formspree at all.
- Button shows an inline loading state during the request (spinner or "Sending…" text + `aria-busy="true"` on the form)

**Success state:**

- Form replaced with a success message (or form hidden and success panel revealed), inside an `aria-live="polite"` region so screen readers announce it
- Success message thanks the user by name (if available), restates response-time expectation, offers the LinkedIn link as an alternate
- Focus moves to the success message heading for keyboard/AT users

**Error states:**

| Scenario | Behavior |
|----------|----------|
| Validation failure (client) | Inline per-field error via existing `showError`/`clearError`; focus moves to first invalid field. Already implemented — do not duplicate. |
| Turnstile missing/expired token | Submit button stays disabled; small inline note next to widget ("Please complete the check") if user tries to submit |
| Network failure / Formspree 5xx | Inline error above form: "Something went wrong. Please try again, or email me directly at [email]." Form stays populated so user can retry without re-typing. |
| Formspree 4xx (rate limit, etc.) | Same inline error; include fallback email link |
| User has JavaScript disabled | Form falls back to native POST (Formspree handles no-JS submissions and redirects to their hosted success page — acceptable for the <1% of users without JS) |

**Anti-spam:**

- **Honeypot:** Hidden `_gotcha` text field. Styled off-screen with `position:absolute; left:-10000px` (not `display:none`, which some bots detect). `tabindex="-1"` and `autocomplete="off"`. Bots fill it → Formspree silently drops the submission.
- **Turnstile:** `<div class="cf-turnstile">` widget with `data-sitekey="{TURNSTILE_SITE_KEY}"`. Load `https://challenges.cloudflare.com/turnstile/v0/api.js` with `async defer`. Use the "managed" mode (Cloudflare decides interactive vs. invisible).

### Section 3: Alternate Channels

- **Section heading:** Short (e.g., "Or reach out directly" — copywriter to refine)
- **LinkedIn:** Link to `https://www.linkedin.com/in/robcparker/` (same URL as footer). `rel="noopener noreferrer"` and `aria-label` indicating new tab, matching the footer pattern.
- **Direct email:** `mailto:` link to the email address confirmed at spec gate (flagged ambiguity — see Approval section). Display the address as visible text so users can copy it; make the whole thing a `mailto:` link.
- **Layout:** Two items side-by-side on desktop, stacked on mobile. Reuse existing card/link primitives from SPEC-002 where they fit; don't fork styles.

### Section 4: Response Expectations

May be merged into the hero intro paragraph rather than rendered as its own section. If kept separate, one short sentence near the form or between sections.

### Security / Privacy

- **PII posture:** Name, email, and message text are submitted to Formspree and delivered to Rob via their service. This is a thin PII surface — no storage in our systems, no analytics on submission content, no replay. Formspree's privacy policy governs the processor relationship. *Arch Review should confirm this is acceptable and does not require disclosure language (privacy note / link to Formspree's policy).*
- **Secret key hygiene:** Turnstile Secret Key stays in Formspree dashboard only; never in repo

- **CSRF:** Not a concern for a public contact form with no authenticated session
- **CSP considerations:** If/when we add a Content-Security-Policy, `challenges.cloudflare.com` and `formspree.io` must be allowed for script-src / connect-src respectively. Noted for the future CSP spec; not required here.

- Alternate Channels: side-by-side desktop, stacked mobile
- Turnstile widget: use default responsive sizing

## Acceptance Criteria

### Page Structure & Content

- **Given** a user lands on `contact.html`, **When** the page renders, **Then** the hero heading, framing paragraph, and contact form are visible above the fold (or within one scroll on mobile)
- **Given** the form section, **When** inspected, **Then** it contains name, email, and message fields, a hidden honeypot, a Turnstile widget, and a submit button
- **Given** the alternate-channels section, **When** rendered, **Then** both a LinkedIn link and a direct email link are present and functional
- **Given** the page renders, **When** the user has JavaScript disabled, **Then** the form still submits via native POST to Formspree (degraded but functional UX)

### Form Validation (Client)

- **Given** the user submits an empty form, **When** submit is clicked, **Then** inline errors appear on all required fields and focus moves to the first invalid field
- **Given** the user enters an invalid email, **When** they blur the email field, **Then** an inline error appears on that field
- **Given** the user enters a valid name/email/message, **When** they blur each field, **Then** no errors are shown and the fields have no `aria-invalid` attribute

### Anti-Spam

- **Given** a bot fills the honeypot `_gotcha` field, **When** the form is submitted, **Then** Formspree silently drops the submission (verified by checking Formspree dashboard does not log it as a successful submission)
- **Given** the Turnstile widget has not produced a token, **When** the user clicks submit, **Then** the submission is blocked (disabled submit button) with an inline hint
- **Given** an expired Turnstile token, **When** the user submits, **Then** the widget re-challenges automatically and submit remains disabled until a fresh token exists

### Submission Flow

- **Given** a valid submission, **When** the form submits, **Then** the submit button shows a loading state and the fetch POST to Formspree occurs with `Accept: application/json`
- **Given** Formspree returns a 200, **When** the response is processed, **Then** the form is replaced by a success message inside an `aria-live` region and focus moves to the success heading
- **Given** Formspree returns an error status, **When** the response is processed, **Then** an inline error message appears above the form with a fallback mention of the direct email, and form field values remain intact
- **Given** the network fails entirely, **When** the fetch rejects, **Then** the same fallback error message appears

### SEO

- **Given** the Contact page is crawled, **When** meta tags are inspected, **Then** `noindex` is NOT present
- **Given** meta tags are inspected, **When** compared to SPEC-004 requirements, **Then** title, meta description, OG title, OG description, og:image, and canonical are all populated with real content

### Integration

- **Given** a user on the Home, About, or Resume page, **When** they click a "Contact" or "Get in Touch" CTA, **Then** they land on the full Contact page (not the placeholder)
- **Given** any page, **When** the Contact nav link is clicked, **Then** the Contact page loads correctly and the active-nav state highlights Contact

### Accessibility

- **Given** a screen reader user, **When** they navigate the form, **Then** every input has an associated label and errors are announced via `aria-describedby` + `aria-invalid`
- **Given** a screen reader user, **When** a successful submission occurs, **Then** the success message is announced via the `aria-live="polite"` region
- **Given** a keyboard-only user, **When** validation fails, **Then** focus moves to the first invalid field without trapping

## Decision Rationale

- **Formspree over a Cloudflare Pages Function:** Lowest-effort option that ships today and decouples from SPEC-000 (Cloudflare Pages setup). Rewriting as a Pages Function is a known migration path at Phase 2 and requires no data migration (Formspree is a pass-through, not a store of record). Chose shipping value now over architectural purity.
- **Turnstile over reCAPTCHA or hCaptcha:** Stack alignment — hosting is already Cloudflare, and Formspree's native Turnstile integration (Nov 2025) means no custom verification code. Turnstile is also privacy-forward (no Google tracking) which matches the site's no-analytics posture.
- **Honeypot AND Turnstile, not one or the other:** Defense in depth is effectively free here. The honeypot catches the dumbest bots without ever hitting the Turnstile network round-trip. Turnstile catches the sophisticated ones.
- **AJAX submission, not native POST with redirect:** Keeps the user on the site; avoids Formspree's branded "Thanks!" page which is off-brand and breaks the hiring-funnel narrative. No-JS fallback preserves the native POST path for the <1% who need it.
- **Minimal fields (name, email, message):** Every additional field is friction at the conversion point. The hiring audience can provide specifics by reply. Revisit if the volume or noise ratio demands triage.
- **No CTAs at the bottom of the page:** This is the terminal page in the funnel. Adding a "Back to Home" or "View Resume" CTA at the bottom would dilute the submit CTA. If the form is submitted successfully, the success state can offer LinkedIn as the one alternate next step.
- **Direct email surfaced publicly:** Accepts a marginal spam risk in exchange for a friction-free fallback for users who dislike contact forms (common among senior execs). The email is already discoverable via LinkedIn.
- **Complex tier, not Standard:** Formspree and Turnstile are both new external API integrations — per `governance/tier-selection-guidelines.md`, that is a mandatory Complex trigger. Also the first architectural pattern for any external service on this site, which is a second Complex trigger. All checklists required.
