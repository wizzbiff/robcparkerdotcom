## Summary

Establish the minimum viable SEO foundation for robcparker.com. The audit found the site is functionally invisible in search — a quoted-domain query returns zero hits from the actual site, and "Rob Parker" is a common-enough name (NASA engineer, Australian dam consultant, Sweetwater sales engineer, wine critic Robert Parker) that disambiguation is non-optional.

Seven concrete additions across all five pages:

1. **Title tags** rewritten on every page with explicit disambiguators (Senior Engineering Executive, AI Platforms, Atlanta).
2. **Meta descriptions** rewritten on every page — 150-160 chars, recruiter-screening voice.
3. **Open Graph tags** present on every page with absolute `og:url` plus a new branded **OG image** (1200×630, name + title + one differentiator) replacing the headshot reuse currently in place.
4. **Twitter Card tags** added on every page (`summary_large_image` referencing the OG image).
5. **Person schema (JSON-LD)** added to every page — `name`, `jobTitle`, `image`, `url`, `sameAs` array linking to LinkedIn. (GitHub URL added to `sameAs` by the future spec promoted from `specs/backlog.md` when R4's GitHub link lands.)
6. **`/sitemap.xml`** generated at project root and committed to the repo. Verified that Cloudflare Pages does NOT auto-generate one (see Decision Rationale).
7. **`/robots.txt`** generated at project root and committed. Allows full crawling and points to the sitemap.

This is the second-largest SEO change after the foundational metadata that shipped with SPEC-001-006. After SPEC-013, the inbound-link acquisition work in `OPERATOR-TODOS.md` (LinkedIn About URL, GitHub bio, etc.) becomes Rob's manual follow-on; that work is what actually drives indexing.

## Context

The audit (`robcparker_com_audit.md` Dimension 6, lines 189-216) is direct: "the site is not indexed, has no inbound links, or both." Quoted searches for `"robcparker.com"` return zero from the actual site. A search for "Rob Parker SugarCRM engineering Atlanta" surfaces LinkedIn, TheOrg, RocketReach — not the site. Action item #4 lists what to do.

Current state of metadata on the site (verified 2026-04-25):

- **Title tags exist on all five pages**, but they are short and lack the strongest disambiguators. E.g., `index.html:8` is "Rob Parker — Senior Engineering Executive" (good but missing AI Platforms / Atlanta). `resume.html:8` is "Rob Parker Resume — Senior Engineering Executive" (similar).
- **Meta descriptions exist** (150-ish chars) but are not consistently calibrated for recruiter-screening voice. Most are accurate; few are sharp.
- **Open Graph tags** are present and well-formed — absolute `og:url`, `og:type=website`, `og:image` set to the headshot at `images/rob-parker-headshot@2x.jpg`. Pattern was established in SPEC-002.
- **Twitter Card tags are NOT present** on any page (verified by grep).
- **JSON-LD / Person schema is NOT present** on any page (verified by grep).
- **`/sitemap.xml` does NOT exist** at project root (verified by `ls`).
- **`/robots.txt` does NOT exist** at project root (verified by `ls`).
- **Canonical tags exist** on most pages (e.g., `resume.html:13`, `contact.html:27`, `advisory.html:24`) but `index.html` and `about.html` are missing canonical — flag for inclusion in this spec.

The audit's framing: "two or three inbound links from properties Google trusts" + per-page metadata + Person schema is the minimum viable SEO foundation. The inbound links (LinkedIn About, GitHub bio, etc.) are operator-side and tracked in `OPERATOR-TODOS.md`. SPEC-013 owns the in-repo half.

## Decision Rationale

- **Why all seven items at once instead of splitting:** SEO foundation is most defensible when it ships as a coherent unit. Splitting (e.g., metadata-only first, schema later) leaves the site half-instrumented for a window during which Google's first crawl after inbound links arrive could form a weak initial impression. One spec, one deploy.
- **Why `/sitemap.xml` is a static file, not generator output:** Cloudflare Pages does not auto-generate a sitemap (verified — see Stack Quirks note below and the WebSearch result documented in Spec Gate context). The site has no build process per CLAUDE.md ("No build process") so adding a generator dependency would violate the project's zero-dependency posture. A static, committed `/sitemap.xml` is the simplest viable approach — five URLs, regenerated by hand when pages are added or removed. SPEC-009-tier "Stack Quirks" precedent.
- **Why a new branded OG image (instead of reusing the headshot from SPEC-002):** the headshot at 800x800 is a square portrait — when LinkedIn/Slack/X render an OG card from a square image, the framing crops badly and the executive presentation suffers. The audit (line 206) explicitly recommends a 1200×630 branded card with name + title + one differentiator. This is a graphic-artist deliverable.
- **Why Person schema specifically (and not Service / Article / etc.):** Person schema is the only structured-data type that materially helps disambiguate Rob from other Rob Parkers. The audit (line 207) makes the case: `sameAs` linking to LinkedIn gives Google a structured signal that this site and the LinkedIn profile are the same person. Service schema would advertise advisory — which contradicts SPEC-011's positioning. Article schema doesn't apply (no articles). Person is the single right choice.
- **Why robots.txt despite Cloudflare's defaults:** Cloudflare Pages serves files from the root of the deployment. Without an explicit `/robots.txt`, the site implicitly allows all crawling (which is what we want) but offers no `Sitemap:` pointer. Adding `/robots.txt` is one HTTP fetch and gives Google's crawler a direct sitemap reference.
- **Why explicit titles per page (not site-wide template):** the site has no build process and no templating engine. Each page's `<title>` lives in its own HTML file. SPEC-013 specifies the exact title for each of the five pages; implementer mirrors them.
- **Why marketing-copywriter pass for titles + meta descriptions:** these are the most-read pieces of micro-copy on the entire site (every search result snippet shows the title and description). They deserve a copywriter pass for tone fit and disambiguator density. Drafts are in this spec; copywriter polishes for voice.
- **Trade-off accepted:** the OG image is a multi-day deliverable if Rob does not have one designed already. SPEC-013 is structured so that the OG image work runs in parallel during Arch Review / implementation; if it's not ready by deploy, the existing headshot stays as the fallback OG image. The metadata + schema + sitemap + robots.txt ship regardless.
- **Why no `<meta name="keywords">`:** deprecated; Google has ignored it since 2009. No SEO benefit.

## Requirements

### R1: Title tags

Per-page title tags (≤60 chars each, marketing-copywriter pass at implementation):

| Page | Recommended title (≤60 chars) | Char count |
|------|-------------------------------|------------|
| `index.html` | `Rob Parker — Senior Engineering Executive, AI Platforms, Atlanta` | 64 (exceeds 60 by 4 — see note) |
| `about.html` | `About Rob Parker — Engineering Executive, AI Platforms, Atlanta` | 63 (exceeds 60 by 3) |
| `resume.html` | `Rob Parker Resume — Senior Engineering Executive, Atlanta` | 56 |
| `contact.html` | `Contact Rob Parker — Engineering Leadership Roles, Atlanta` | 57 |
| `advisory.html` | `Technical Advisory — Rob Parker, Senior Engineering Executive` | 60 |

**Note on title length:** the 60-char target is a Google search-result-display heuristic, not a hard limit. Titles are not truncated at byte boundaries — Google uses pixel width. The recommended titles above are at-most ~64 chars. Marketing-copywriter is empowered to tighten; the disambiguators (Engineering Executive / AI Platforms / Atlanta) are non-negotiable on the home and about pages because that's where the disambiguation work happens. Resume and Contact can be shorter because they're page-specific.

**Why these disambiguators:**
- "Senior Engineering Executive" — Rob's positioning; matches existing site copy and resume.
- "AI Platforms" — the differentiator that separates him from the NASA engineer / wine critic / etc. Per audit line 205.
- "Atlanta" — geographic disambiguator; matches Rob's actual location and is referenced in the resume and About page.

Title ordering puts the brand (Rob Parker) first on home and resume — most natural for a personal-brand site. About and Contact lead with the page label ("About" / "Contact") because those terms are the search query intent for someone navigating to those pages.

### R3: Open Graph tags + branded OG image

**Per-page OG tags (already mostly in place — verify and refine):**

```html
<meta property="og:type" content="website">
<meta property="og:url" content="https://www.robcparker.com/{path}">
<meta property="og:title" content="{matches <title> on the page}">
<meta property="og:description" content="{matches meta description on the page}">
<meta property="og:image" content="https://www.robcparker.com/images/og/og-card.png">
<meta property="og:image:width" content="1200">
<meta property="og:image:height" content="630">
<meta property="og:image:alt" content="Rob Parker — Senior Engineering Executive, AI Platforms, Atlanta">
```

**The branded OG image** is the new asset. Specification:

- **Dimensions:** 1200×630 (Open Graph standard).
- **Format:** PNG (lossless, broad support across crawlers). JPEG is acceptable if file-size is a concern; PNG preferred.
- **Filename / path:** `/images/og/og-card.png` (committed). Source PSD/SVG/Figma master in `images/source/og/` (gitignored per CLAUDE.md image convention).
- **Content:** name ("Rob Parker"), positioning ("Senior Engineering Executive · AI Platforms · Atlanta"), and one differentiator (recommended: a typographic mark or the SPEC-007 favicon glyph, OR the metric "25+ Years" or similar). `graphic-artist` agent designs at implementation.
- **Tone:** matches the SPEC-008/009 design language — Fraunces serif for the name, Inter for positioning, charcoal-on-cream or magenta-on-charcoal palette per `graphic-artist` judgment. Must NOT introduce a new color or font outside the SPEC-008/009 palette.
- **Accessibility:** the `og:image:alt` attribute carries the announced text for screen readers and accessibility-aware crawlers. Required.

**Asset deferral path:** if the branded OG image is not ready by Arch Review, R3 ships with the existing headshot (`https://www.robcparker.com/images/rob-parker-headshot@2x.jpg`) as a placeholder, with a follow-on trivial spec to swap in the branded card. The placeholder swap is mechanical (one URL change × five HTML files).

**Per-page OG copy uniqueness:** each page's `og:title` and `og:description` mirror that page's `<title>` and meta description respectively (R1, R2). The audit specifically says (line 206) "every page" should have OG tags, not just the home page.

### R4: Twitter Card tags

```html
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:title" content="{matches og:title}">
<meta name="twitter:description" content="{matches og:description}">
<meta name="twitter:image" content="{matches og:image}">
<meta name="twitter:image:alt" content="{matches og:image:alt}">
```

`twitter:site` is **omitted** at Rob's direction (Spec Gate Q3) — Rob has no X/Twitter handle to associate. The five tags above are sufficient for the card to render.

The `summary_large_image` card type renders the 1200×630 OG image at full width on X / Twitter (the alternative `summary` card uses a small square image; not what the audit specifies).

### R5: Person schema (JSON-LD)

**Per-page block** inserted in `<head>` after the OG / Twitter tags:

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Person",
  "name": "Rob Parker",
  "jobTitle": "Senior Engineering Executive",
  "description": "Senior engineering executive with 25+ years of experience building AI-native platforms and scaling high-performing teams.",
  "image": "https://www.robcparker.com/images/rob-parker-headshot@2x.jpg",
  "url": "https://www.robcparker.com/",
  "address": {
    "@type": "PostalAddress",
    "addressLocality": "Atlanta",
    "addressRegion": "GA",
    "addressCountry": "US"
  },
  "alumniOf": [
    { "@type": "CollegeOrUniversity", "name": "Georgia Institute of Technology" },
    { "@type": "CollegeOrUniversity", "name": "Clemson University" }
  ],
  "sameAs": [
    "https://www.linkedin.com/in/robcparker/"
  ]
}
</script>
```

**Constraints:**
- Identical block on every page is acceptable (Person schema is page-agnostic — Google de-duplicates). The block IS the same across all five pages. No per-page variation needed.
- `sameAs` ships with **LinkedIn only** at SPEC-013 ship time. SPEC-012 R4 (GitHub link) was deferred at Spec Gate 2026-04-26 to `specs/backlog.md`. When R4's follow-on spec lands, that spec also adds the GitHub URL to the `sameAs` array on every page.
- LinkedIn URL confirmed at `https://www.linkedin.com/in/robcparker/` from existing site usage (e.g., `index.html:248`).
- `image` reuses the existing headshot URL. (The branded OG image is for OG/Twitter, not for Person `image` — Google's Person `image` field expects a portrait of the actual person.)
- `description` matches the most-truthful one-liner from the site's body copy; copywriter may tighten.
- `alumniOf` is included because the audit (Dimension 7, line 240) recommends adding education to the About page; mirroring it in schema gives Google another disambiguation signal.

**Do NOT include:**
- `email` — leaks contact@robcparker.com to scrapers needlessly. The site's contact form is the right intake path.
- `telephone` — same reason.
- `worksFor` — Rob is between roles; no current employer to cite truthfully.
- `knowsAbout` — gimmicky; not load-bearing for disambiguation.

### R6: `/sitemap.xml`

**Static file** committed at project root. Content:

```xml

  </url>
  <url>
    <loc>https://www.robcparker.com/about.html</loc>
    <lastmod>2026-04-25</lastmod>
    <changefreq>monthly</changefreq>
    <priority>0.9</priority>
  </url>
  <url>
    <loc>https://www.robcparker.com/resume.html</loc>
    <lastmod>2026-04-25</lastmod>
    <changefreq>monthly</changefreq>
    <priority>0.9</priority>
  </url>
  <url>
    <loc>https://www.robcparker.com/contact.html</loc>

    <priority>0.7</priority>
  </url>
  <url>
    <loc>https://www.robcparker.com/advisory.html</loc>
    <lastmod>2026-04-25</lastmod>
    <changefreq>yearly</changefreq>
    <priority>0.6</priority>
  </url>
</urlset>
```

**Notes:**
- `lastmod` set to deploy date at implementation time. Updated manually whenever a page changes meaningfully (Rob's call; not enforced).
- `priority` is hint-only and minor. Home highest; advisory lowest because per SPEC-011 it's referral-only.
- `advisory.html` IS included in the sitemap. Its delisting (SPEC-011) is a navigation/UX decision, not an indexing decision. The page is real and crawlable; sitemap should list it.
- **No build automation.** Per CLAUDE.md no build process exists; the file is hand-maintained. Trade-off accepted.

### R7: `/robots.txt`

**Static file** committed at project root. Content:

```
User-agent: *
Allow: /

Sitemap: https://www.robcparker.com/sitemap.xml
```

**Notes:**
- Permits all crawlers everywhere. The site has no admin pages, no auth, no PII, no draft content.
- The `Sitemap:` line is the load-bearing piece — it tells crawlers where to find `/sitemap.xml`.
- No `Disallow:` rules. Adding spurious `Disallow:` rules is a common SEO foot-gun.

### R8: Canonical tags audit

`index.html` and `about.html` are missing `<link rel="canonical">` (verified). Add to both:

- `index.html`: `<link rel="canonical" href="https://www.robcparker.com/">`
- `about.html`: `<link rel="canonical" href="https://www.robcparker.com/about.html">`

The other three pages already have canonicals (resume, contact, advisory). Verify those are correct and not changed by this spec.

### R9: Stack quirks update

`governance/stack-quirks.md` gets a new entry under a new "## SEO" section, capturing what was learned:

> **Cloudflare Pages does NOT automatically generate `sitemap.xml` or `robots.txt`.** Static sites on Cloudflare Pages must commit both files at project root (or generate them at build time, but this site has no build). Cloudflare Pages adds an `X-Robots-Tag: noindex` header on preview URLs only — production deploys are crawlable by default. Established in SPEC-013.

This entry locks in the verified third-party behavior so future specs do not re-research the same question.

## Non-Functional Requirements

### Performance

- New `<head>` content adds ~1-2KB per page (OG + Twitter + JSON-LD + canonical). Negligible.
- OG image is a single new asset (~150-300KB at 1200×630 PNG with text + minimal graphics — `graphic-artist` optimizes). Loaded only by social-media crawlers, not by visitors.
- `/sitemap.xml` and `/robots.txt` are tiny static files. Zero rendering cost.

### Accessibility

- `og:image:alt` and `twitter:image:alt` attributes are populated.
- JSON-LD is invisible to screen readers and visual users; no a11y impact.
- No new visible content.

### SEO

- This IS the SEO spec. Every requirement above directly serves SEO.
- Net expected impact: Google indexes the site within ~1-2 weeks of deploy + first inbound link, vs. the current "functionally invisible" state.
- No new `noindex` introduced. No `nofollow` introduced. Site is fully crawlable.

### Responsive

- N/A — no visual changes.

## Acceptance Criteria

### R1 — Titles

- **Given** every HTML page is loaded, **When** the `<title>` tag is inspected, **Then** the title matches the per-page recommended title (or the marketing-copywriter-tightened equivalent within the disambiguator constraint).
- **Given** any page's title is read in a search-result preview, **When** truncated to ~60 chars, **Then** at least one disambiguator (Engineering Executive / AI Platforms / Atlanta / Resume / Contact / Technical Advisory) is visible.

### R2 — Meta descriptions

- **Given** every HTML page is loaded, **When** `<meta name="description">` is inspected, **Then** the description matches the per-page recommended description (or copywriter-tightened equivalent).
- **Given** the `<meta name="description">` content, **When** char-counted, **Then** every description is between 140 and 160 chars.

### R3 — Open Graph

- **Given** every HTML page, **When** OG tags are inspected, **Then** all of `og:type`, `og:url` (absolute), `og:title`, `og:description`, `og:image` (absolute), `og:image:width`, `og:image:height`, `og:image:alt` are present.
- **Given** the OG image at the URL, **When** fetched, **Then** the file exists at `/images/og/og-card.png`, is 1200×630, and is a non-trivial image (not a placeholder gray box). If the branded card is deferred, the existing headshot URL is used and the deferral is documented at QA Gate.
- **Given** an LinkedIn / X / Slack share preview, **When** the page URL is dropped in, **Then** the card renders with the OG image, title, and description (verified post-deploy via opengraph.xyz — operator todo).

### R4 — Twitter Card

- **Given** every HTML page, **When** Twitter Card tags are inspected, **Then** `twitter:card` (summary_large_image), `twitter:title`, `twitter:description`, `twitter:image`, and `twitter:image:alt` are present.

### R5 — Person schema

- **Given** every HTML page, **When** the JSON-LD `<script type="application/ld+json">` block is inspected, **Then** it is valid JSON, parses successfully, and contains all of `@context`, `@type=Person`, `name`, `jobTitle`, `description`, `image`, `url`, `address`, `alumniOf`, `sameAs`.
- **Given** the JSON-LD block, **When** validated against `https://search.google.com/test/rich-results`, **Then** zero errors and zero warnings on critical fields. (Operator todo to actually run the test post-deploy.)
- **Given** SPEC-012 R4 was deferred at Spec Gate, **When** SPEC-013 ships, **Then** the `sameAs` array contains the LinkedIn URL only. (A follow-on spec adds GitHub to `sameAs` when R4's backlog item is promoted.)

### R6 — Sitemap

- **Given** the implementer runs `ls /sitemap.xml` from the project root, **When** the spec is implemented, **Then** the file exists.
- **Given** `/sitemap.xml` is parsed as XML, **When** the parser runs, **Then** zero errors and exactly five `<url>` entries (home, about, resume, contact, advisory) with absolute `<loc>` URLs.
- **Given** the sitemap is fetched via `curl https://www.robcparker.com/sitemap.xml` post-deploy, **When** the response is inspected, **Then** the response code is 200 and the Content-Type is `application/xml` or `text/xml`.

### R7 — Robots

- **Given** the implementer runs `ls /robots.txt`, **When** the spec is implemented, **Then** the file exists.
- **Given** `/robots.txt` content, **When** read, **Then** it contains `User-agent: *`, `Allow: /`, and `Sitemap: https://www.robcparker.com/sitemap.xml`.
- **Given** the robots.txt is fetched via `curl https://www.robcparker.com/robots.txt` post-deploy, **When** the response is inspected, **Then** the response code is 200 and the file contents match.

### R8 — Canonicals

- **Given** every HTML page, **When** `<link rel="canonical">` is inspected, **Then** every page has a canonical and it points to the absolute URL of that page.

### Cross-cutting

- **Given** the deployed site, **When** Lighthouse SEO audit is run (operator todo), **Then** the score is 90 or higher per audit line 215.
- **Given** any page's `<head>`, **When** read end-to-end, **Then** there is no duplicate of any meta property (e.g., one `og:title` per page, one canonical per page).

## Tier Selection — Standard

**Tier:** Standard. Per `governance/tier-selection-guidelines.md`:

| Trigger | Applies? | Notes |
|---------|----------|-------|
| Authentication / authorization changes | No | None |
| Payment or financial data | No | None |
| PII / PHI handling | No | None |
| New external API integration | No | None |
| Database schema change | No | None |
| Core domain model modification | No | Metadata only |
| Framework or platform migration | No | None |
| First implementation of new architectural pattern | **Borderline** — Person schema (JSON-LD) is the first time structured data is added. But it's a one-time addition of a well-known pattern, not a new architectural decision. Treated as additive content, not architectural. |

**Decision flow:**
1. Does it change any code paths? Yes — `<head>` on five pages, two new root-level static files.
2. Does it follow existing patterns? Mostly yes (titles, meta, OG already in place; new Twitter Card and JSON-LD are conventional). Limited scope.
3. Does it affect multiple components? Yes — every page. But the changes are mechanical and identical across pages. Standard.
4. Does it introduce new architectural patterns? JSON-LD is new but is content, not architecture. Sitemap.xml is new but is a static file, not a pattern.

**Tier rationale:** Standard. Touches all five HTML pages plus two new root files. Includes one third-party feature claim (Cloudflare Pages sitemap behavior) that has been verified. Includes a `graphic-artist` deliverable (branded OG image) with explicit deferral path. Includes content drafts (titles, meta descriptions) that need marketing-copywriter polish.
